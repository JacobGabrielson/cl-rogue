# Model-Driven Monster AI

## Overview

The original Rogue monster AI is a pure greedy pathfinder: every monster
that is "running" (ISRUN flag set) calls `do-chase` each turn, which
picks the adjacent cell that minimises Manhattan distance to the target.
Confused monsters move randomly; Invisible Stalkers and Bats have a
higher confusion probability. That's it.

The goal of this project is to replace (or optionally augment) that
fixed pathfinder with a small learned model so monsters feel meaningfully
smarter — flanking, retreating when hurt, coordinating with other nearby
monsters, etc. — without requiring any external service at runtime or
during training.

The pipeline is:

1. **Generate training data** — run many self-play episodes with the
   headless Python driver and label each monster's move using a
   hand-coded expert heuristic (A\* pathfinding + simple tactical rules).
   No API calls, no internet connection needed.

2. **Train a small model** — fit a compact classifier on those
   (state, action) pairs. The model learns to generalise the heuristic's
   behaviour across varied game states.

3. **Hook it in** — replace (or wrap) `do-chase` / `chase` in
   `chase.lisp` with a call to the model for any monster that has a new
   `ISMODEL` flag set.

The reason to bother training a model rather than just running the Python
heuristic directly is that the model lives inside cl-rogue as a tiny
serialised file and requires no Python subprocess at play time. It can
also be retrained to exhibit different behaviours (more aggressive,
type-specific personalities, etc.) without changing any Lisp code.

## State Representation

Each monster decision is represented as a fixed-size feature vector
assembled from the game state visible to that monster at the moment of
its turn.

### Local grid window
A 9×9 tile window centred on the monster (81 cells). Each cell is
one-hot encoded over the following tile classes:

| Symbol | Class |
|--------|-------|
| ` ` (space / unseen) | 0 |
| `.` floor | 1 |
| `#` passage | 2 |
| `\|` / `-` wall | 3 |
| `+` door | 4 |
| `@` player | 5 |
| letter A–Z (other monster) | 6 |
| item (`:!?*)]/=/>`) | 7 |

That gives 81 × 8 = **648 bits** for the local map. A local window is
sufficient for the tactical decisions that matter most (step toward
player, avoid walls, don't walk into another monster). Long-range
coordination can be added later as extra scalar features rather than
by expanding the grid.

### Monster state (scalar features)
- Monster type (26 one-hot values, A–Z)
- Current HP fraction (0–1)
- Monster level (normalised)
- Distance to player (normalised by map diagonal)
- Player visible? (bool)
- Player HP fraction (0–1, from status line — monsters should press harder
  when the player is weak)
- Number of other monsters in the same room
- Direction to nearest ally (dx, dy, normalised) — enables basic flanking
- Flags: ISHASTE, ISSLOW, ISBLIND, ISHUH, ISMEAN, ISGREED (6 bits)

Total input size: ≈ **700 floats** (sparse; most grid cells are 0).

### Action space
Nine discrete actions (same as the rogue movement grid):

```
7 8 9       y k u
4 5 6  →    h . l
1 2 3       b j n
```

`5` (stay) is included so the model can choose to hold position (e.g.
blocking a corridor while another monster flanks).

## Training Data Generation

All training data is generated by self-play: the Python driver runs
episodes and a hand-coded expert heuristic labels each monster move.
No external API is required.

### Expert heuristic (`driver/expert.py`)

The expert produces a move for each monster by applying a small set of
prioritised rules:

1. **Attack** — if the player is adjacent, move onto the player's cell.
2. **Retreat** — if HP < 25 % and the player is visible, move away (pick
   the adjacent cell that maximises distance to player).
3. **Flank** — if another monster is in the same room and closer to the
   player, move to approach the player from the opposite side (pick the
   walkable cell that maximises separation from the ally while still
   reducing distance to player).
4. **Chase** — A\* shortest path to the player through walkable tiles.
5. **Wander** — if player is not visible, move randomly toward room exits.

These rules produce strictly better behaviour than the greedy pathfinder:
the retreat and flank rules in particular create emergent pack tactics
that the original AI never exhibits.

### Collection script (`driver/collect_training_data.py`)

```
for each episode:
    start game, wait for dungeon to render
    for each turn:
        parse snapshot: tile grid, monster positions, player position/HP
        for each running monster M:
            build feature_vector from 9×9 window + scalar features
            action = expert.choose(M, game_state)
            record (feature_vector, action_index) to JSONL
        send player move (heuristic: explore, pick up items, descend)
    until player dies or level 5 reached
```

The player's moves are also heuristic (not random) so the monsters
encounter realistic situations — corridors, multi-room chases, combat
at low HP — rather than a stationary target.

Target corpus size: **20,000–50,000 (state, action) pairs**. This is
much smaller than an LLM-labelled corpus because the expert is
deterministic and consistent; less data is needed to fit it.

### Quality checks
- Assert every recorded action is a legal move (walkable tile). The
  expert must never generate an illegal action; any that slip through
  indicate a bug in the expert, not noise to filter.
- Stratify by monster type and dungeon level so rare high-level monsters
  are represented.

## Model Architecture

### Option A — Gradient-boosted trees (XGBoost / LightGBM)
- Pros: trains in seconds on 50k examples, serialised size ~500 KB, no
  GPU needed, trivially callable from Python.
- Cons: no spatial inductive bias; the 9×9 grid is flattened.
- **Start here.** Given that the expert heuristic is rule-based and
  consistent, a tree ensemble should fit it very well.

### Option B — Small CNN + MLP
```
Input: 9×9×8 one-hot grid  →  Conv2d(8→16, 3×3) → ReLU → Conv2d(16→32, 3×3)
       → flatten → concat scalar features → FC(256) → FC(9, softmax)
```
- Pros: spatial locality is modelled explicitly; may generalise better
  to states that are rare in training.
- Cons: needs PyTorch (~200 MB), slightly more complex training loop.
- Switch to this if XGBoost tops out below ~80 % top-1.

### Option C — Distilled decision tree
Train Option A first, then distil into a shallow decision tree (depth
~15) for zero-dependency inference translatable to pure Lisp if desired.

**Recommended starting point:** Option A (XGBoost).

## Integration into cl-rogue

### New monster flag: `ISMODEL`
Add `(defconstant ISMODEL #o0200000)` to `rogue.lisp`. Set it on some or
all monsters at spawn time (configurable, defaulting to the "harder"
monster types from level 5 onward).

### Inference sidecar process
Rather than embedding Python inside SBCL, run a small server
(`driver/model_server.py`) that loads the model once and serves requests
over a Unix socket.

On each monster turn where `ISMODEL` is set, cl-rogue calls a new Lisp
function `model-move` that:
1. Serialises the feature vector as a flat binary blob (fixed-size array
   of 32-bit floats, little-endian).
2. Sends it to the model server via a Unix socket write.
3. Reads back the action index (single byte, 0–8).
4. Maps the action to a `coord` delta and passes it to the existing move
   machinery (bypassing `chase`).

Round-trip should be < 1 ms over a local socket.

### Fallback
If the model server is not running, `model-move` silently falls back to
the original `do-chase` logic. The game is always playable without the
AI component.

### New source files
| File | Purpose |
|------|---------|
| `driver/expert.py` | Hand-coded A\* + tactical rules heuristic |
| `driver/collect_training_data.py` | Self-play episode runner |
| `driver/train_model.py` | Feature engineering + XGBoost training |
| `driver/model_server.py` | Inference server (Unix socket) |
| `driver/requirements-ai.txt` | Extra deps (xgboost, numpy, scipy) |
| `model-move.lisp` | Lisp client for the inference server |

`model-move.lisp` is added to `cl-rogue.asd` (after `chase.lisp`).

## Phased Implementation Plan

### Phase 1 — Expert heuristic + data collection
- [ ] Write `expert.py`: A\* on the tile grid, plus retreat/flank rules.
- [ ] Write `collect_training_data.py`: drives episodes, calls expert,
      writes JSONL to `training_data/`.
- [ ] Assert 100 % legal-move rate on a 1,000-example sample.
- [ ] Collect 5,000 examples to prove out the pipeline.

### Phase 2 — Feature engineering & baseline model
- [ ] Write `featurise.py`: reads JSONL, builds 700-float vectors,
      splits train/val/test, saves as numpy `.npz`.
- [ ] Train XGBoost; report top-1 accuracy. Target: > 80 % (the expert
      is deterministic so this should be achievable).
- [ ] Confusion matrix by monster type to spot any poorly-learned cases.

### Phase 3 — Inference server
- [ ] Write `model_server.py`: loads model, listens on
      `/tmp/cl-rogue-model.sock`, decodes float array, returns action byte.
- [ ] Benchmark round-trip latency; must be < 5 ms p99.

### Phase 4 — Lisp integration
- [ ] Add `ISMODEL` flag and `model-move.lisp`.
- [ ] Wire into `do-chase`: if `(on th ISMODEL)`, call `model-move`.
      Fall back to original `chase` if socket unavailable.
- [ ] Add a wizard-mode command (`^M`?) to toggle model AI on/off at
      runtime for comparison.
- [ ] Playtest on dungeon levels 1–5.

### Phase 5 — Iteration
- [ ] Scale to 50k examples and retrain.
- [ ] Tune the expert's tactical rules based on playtest observations.
- [ ] Try CNN (Option B) if desired for better generalisation.
- [ ] Experiment with per-monster-type training (separate model per
      monster family) vs a single shared model with type as a feature.

## Open Questions

1. **How smart should the model be?** The expert heuristic is already
   meaningfully better than the original greedy AI. If every monster
   plays close to optimally the game may become too hard. Options: sample
   from the softmax (temperature > 1) to add variance; only apply
   `ISMODEL` to higher-level monsters; introduce a per-monster-type
   difficulty dial.

2. **Player modelling?** Adding `player_hp_fraction`, `player_is_confused`,
   `player_has_bow` as scalar features is low-cost and would let the
   model learn to press harder when the player is weak.

3. **Serialisation format for the Lisp↔Python socket?** A flat array of
   32-bit IEEE-754 floats (little-endian, fixed size) is simplest and
   fastest. A length-prefixed JSON blob is ~5× larger but much easier to
   debug during development; switch to binary once the protocol is stable.

4. **Skip the model entirely?** If the expert heuristic is good enough
   on its own, we could run it as a permanent Python sidecar instead of
   training a model. The model is worth it mainly for zero-dependency
   play (no Python required once trained) and the ability to tweak
   behaviour by retraining rather than editing rules.
